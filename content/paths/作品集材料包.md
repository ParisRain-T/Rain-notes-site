# 作品集材料包（让面试官“快速信任你”）

你已经在做 P0→P1 这种很强的工程主线。真正拉开投递转化率差距的，不是再学一堆知识点，而是把成果变成**可复现、可解释、可追问**的材料包。

> 规格以 [[P0_SPEC_v1]] 为准；执行节奏以 [[执行路径]] 为准。

## 1）最终交付清单（Release DoD）

P0/P1 每次发版都用同一套 DoD（Definition of Done）审计：

- `docker compose up -d` 一键启动（写在 README 的 Quickstart）
- 至少 3 个验收脚本全 PASS（串口/MQTT/API）
- 连续运行 ≥ 24h（后期目标 48h），并写稳定性报告（掉线次数、重连耗时、失败原因 Top3）
- 英文 README 完整：Architecture + API + Troubleshooting
- demo 视频 2–4 分钟（能从零演示闭环）

## 2）简历两版 bullet（同一项目两种叙述）

### 2.1 嵌入式 / 机器人系统岗（端侧重点）

把项目写成“工程能力 + 调试能力”：

- 基于 STM32（F411/F103）实现 UART/I2C/Timer/PWM 驱动与模块化分层（`bsp/drivers/app`），完成传感采集与执行器控制
- 设计并实现二进制串口协议（SOF/LEN/CRC16/ACK），使用 ring buffer + parser 状态机解决丢包与坏帧重同步问题
- 实现命令幂等（`cmd_id` 去重）与超时重试策略，保障 QoS1 重复投递下不重复执行
- 建立可观测链路：统一日志字段（`device_id/cmd_id/seq`）与错误码表，将定位时间从 X 分钟优化到 Y 分钟（写真实数字）

### 2.2 IoT 平台 / 边缘网关 / 后端岗（平台重点）

把项目写成“端到云链路 + 交付能力”：

- 搭建 MQTT Broker + FastAPI + Postgres 的端到云数据链路，实现遥测入库、历史查询与指令下发 API，并固化 Topic/消息格式规范
- 基于 Docker Compose 实现一键部署与复现，配套端到端验收脚本与故障注入 runbook，保证弱网/重启场景可恢复
- 构建命令状态机（CREATED/SENT/ACKED/TIMEOUT/FAILED）与 ACK 回执入库，实现端云状态一致性与可追踪审计

## 3）面试追问清单（建议至少 20 题）

你需要提前准备“追问→你的答法要点”，避免现场临时编故事。

### 协议/固件

1. 为什么要二进制协议？为什么不用纯文本/JSON？
2. CRC 覆盖范围怎么定？为什么是 CCITT-FALSE？
3. parser 如何 resync？坏帧会不会导致卡死？
4. ISR 为什么只入队不解析？ring buffer 容量怎么估算？
5. `HAL_Delay` 有什么问题？你怎么做周期任务？

### MQTT/可靠性

6. QoS0/1/2 区别？为什么 cmd/ack 用 QoS1？
7. QoS1 重复投递怎么处理？幂等怎么保证？
8. retain 适合用在哪里？有哪些坑？
9. 断网/重连策略怎么做？固定重试还是退避？
10. broker 重启会发生什么？你的系统如何恢复？

### 后端/DB

11. telemetry 表怎么建索引？按什么查询模式设计？
12. commands 表的状态机为什么这么设计？
13. API 设计如何兼顾同步等待 ACK 与异步轮询？
14. 如何保证“命令下发成功但 ACK 丢失”时状态一致？

### 可观测/排障

15. 你遇到过最难的 bug 是什么？怎么定位的？
16. 如果命令失败，你如何在 1 分钟定位是 MCU/网关/broker/后端哪一层？
17. 你记录了哪些关键日志字段？为什么？

### 安全（最小线）

18. 不做 TLS 时你如何做最小鉴权/防误触发？（token/HMAC/白名单）
19. 如何防止重放命令？（cmd_id/窗口/过期时间）

### P1（小车/机器人）

20. 安全策略是什么？（急停、命令超时刹车、限速、低电压保护）
21. 里程计/坐标系怎么定义？时间戳如何一致？

## 4）图与讲稿（面试官最爱）

### 4.1 三张图（建议每个项目必备）

1. 架构总图（端/边/云模块与数据流）
2. 数据流图（telemetry 路径）
3. 异常流图（断网、重启、重复命令、超时）

### 4.2 英文讲稿（模板）

#### 90 秒版（电梯稿）

1. What: I built an end-to-cloud IoT system with an MCU device, a gateway, MQTT, a FastAPI backend, and PostgreSQL.
2. Why: To practice system engineering, reliability, and observability in a real closed-loop project.
3. How: Telemetry goes up, commands go down, and every command is acknowledged and stored with idempotency guarantees.
4. Result: One-command startup with Docker Compose, automated end-to-end tests, and a demo video showing the full loop.

#### 5 分钟版（结构）

1. 背景与目标（为什么做、要证明什么能力）
2. 架构（端/边/云，Topic/协议）
3. 关键难点与取舍（QoS、幂等、超时重试、可观测）
4. 一个真实故障复盘（怎么定位、怎么修、怎么避免复发）
5. 指标与结果（成功率、稳定性时长、复现耗时）
6. 下一步（Backlog：最小安全线、P1 升级、ROS2 可选）

## 5）内容输出复用（Obsidian → Quartz → 短视频）

你不需要做“自媒体工程”，只要把**项目周报**固定下来，就能自动生成博客与视频素材。

### 5.1 每周项目周报（固定 5 段）

1. 本周目标（1 句话）
2. 实现了什么（3 条）
3. 最大卡点（1 条）
4. 我怎么定位/解决（步骤）
5. 下周计划（2 条）

### 5.2 60 秒短视频脚本（三段式）

- 5 秒：钩子（“我调了 3 天 STM32 调试，原来问题在这一步”）
- 40 秒：关键知识点 + 演示（比如：串口协议/ACK/重试，或一次真实排障）
- 15 秒：复盘 + 下期预告（“下周我把命令幂等做成 `cmd_id` 去重”）
